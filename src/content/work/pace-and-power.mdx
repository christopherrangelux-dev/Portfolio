## The Problem

Hybrid athletes — people who lift heavy and run serious mileage in the same week — face a fatigue problem that generic training apps ignore. Tools like Strava and Strong track one modality well and are blind to the other. The conflict shows up in the body: CNS load from a heavy lower-body lift on Thursday and structural load from a run two days later don't reset on the same clock.

Placing them too close without flagging the overlap leaves recovery to guesswork. This tool makes the cost of each session visible and flags when back-to-back days are likely to compound.

## Architecture Notes

The collision engine runs entirely client-side — no server, no model, no training data. It assigns a base cost to each session type, applies recovery modifiers from the sliders, then checks adjacent days against a rule table.

**Cost lookup:**

| Session | Base cost |
|---------|-----------|
| Lower body heavy lift | CNS 3 |
| Lower body moderate lift | CNS 2 |
| Upper body lift | CNS 1 |
| Long run (6+ mi) | Structural 3 |
| Tempo / interval | Structural 2 |
| Easy / Zone 2 under 5 mi | Structural 1 |
| Stairmaster / incline walk | Structural 1 |

**Collision rules (evaluated in order, first match wins per pair):**

| Rule | Condition | Severity |
|------|-----------|----------|
| Critical overlap | Day N ≥ 5 AND Day N+1 ≥ 4 | Danger |
| Elevated overlap | Day N ≥ 4 AND Day N+1 ≥ 3 | Warning |
| Back-to-back lower | Both days have a lower-body lift | Warning |
| Everything else | — | OK |

**Recovery modifiers:** Sleep below 6 → +1 to combined cost. Systemic stress above 7 → +1 to all CNS costs. Deload week → halve all base costs before modifiers.

The rule table is intentionally legible. Every assumption is inspectable. The tool is wrong in known ways — it doesn't account for training age, accumulated weekly volume, or individual recovery variance — and those limits are visible rather than hidden inside a model.

**Implementation:** All state lives in module-scoped TypeScript variables inside a single `.astro` component. DOM updates go through `.dataset`, `.style.width`, and `.textContent` — no framework, no virtual DOM, no library dependencies. Event binding fires once inside `astro:page-load` with a window-flag guard to handle Astro's view transitions cleanly.

## Reflection

The design decision I kept coming back to: should recovery context (sleep, stress, deload) modify the displayed scores or just the collision flags? I landed on modifying the scores themselves, because it makes the feedback loop legible — you can see why a session changes color, not just that it did.

What's next: user-defined sessions (the matrix is currently read-only), a 4-week rolling view to track accumulated load across mesocycles, and HRV input as a recovery signal that updates the sleep slider automatically.
