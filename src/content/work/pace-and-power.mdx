## The Problem

Every Thursday I hit heavy squats, hip thrusts, and Romanian deadlifts. Then Friday I ran. Some weeks I felt fine. Others, my legs were wrecked before I hit mile one. I couldn't tell in advance which kind of week it was going to be, because I had no model for how training days compound.

Strava tracks runs. Strong tracks lifts. Neither knows the other exists. The fatigue cost of Thursday's squat session doesn't factor into Friday's run plan. You're guessing.

I apply systems thinking at work to make invisible things visible: permissions, debt, git history. This was the same problem applied to my own body. I wanted to see the cost of each session before committing to it, and to know when two adjacent days were likely to stack in ways I couldn't absorb.

## Live Tool

Pre-loaded with my actual Thu–Sun training block. Every session is editable: swap Thursday from Lower Heavy to Upper body and watch the collision strip disappear. Change a rest day to a long run and see what it does to the week. The recovery sliders let you adjust for sleep quality, systemic stress, and deload weeks.

## The Design Decision

The core question was whether to make the engine rule-based or model-based. A trained model would be more accurate in ways I couldn't enumerate. A rule table is accurate in ways I can.

Every cost in the table below is inspectable and challengeable. When the system flags a warning between Friday and Saturday, you can see exactly why: the Lower B lift carries a CNS cost of 3, the easy run adds structural cost 1, and back-to-back lower body days trigger the warning rule. The assumptions aren't hidden inside a score. That makes the tool wrong in legible ways, which is more honest than being wrong in opaque ones.

**Cost lookup:**

| Session | Base cost |
|---------|-----------|
| Lower body heavy lift | CNS 3 |
| Lower body moderate lift | CNS 2 |
| Upper body lift | CNS 1 |
| Long run (8+ mi) | Structural 3 |
| Tempo run | Structural 2 |
| Easy / Zone 2 run under 5 mi | Structural 1 |
| Stairmaster / incline walk | Structural 1 |

**Collision rules, applied in order (first match wins per pair):**

| Condition | Severity |
|-----------|----------|
| Day N load ≥ 5 AND Day N+1 load ≥ 4 | Danger |
| Day N load ≥ 4 AND Day N+1 load ≥ 3 | Warning |
| Both days have a lower-body lift | Warning |
| Everything else | OK |

**Recovery modifiers:** Sleep below 6 adds +1 to combined cost. Systemic stress above 7 adds +1 to all CNS costs. Deload week halves all base costs before modifiers apply.

**Implementation:** All state lives in module-scoped TypeScript variables inside a single `.astro` component. DOM updates go through `.dataset`, `.style.width`, and `.textContent`. No framework, no virtual DOM, no library dependencies. Event binding fires once inside `astro:page-load` with a window-flag guard for Astro's view transitions.

## Reflection

The same instinct that shows up here shows up across my work: when something is opaque, I reach for structure to make it legible. Branch Theory does it for Git history. Evidence Engine does it for A/B test logic. Nexus Ops does it for permissions. This does it for my own body.

The decision to modify the fatigue scores themselves rather than just the collision flags was the most deliberate call in the build. If sleep drops below 6 and Thursday's cost jumps from 4 to 5, you see the card turn red. You know why it changed. That feedback loop is the point.

What I'd build next: a 4-week rolling view to track accumulated load across mesocycles, user-defined session names, and HRV as a live recovery signal feeding the sleep slider automatically.
